混合检索：Dense Embedding + BM25

在企业级 RAG（Retrieval-Augmented Generation，检索增强生成）系统的落地过程中，单纯依赖密集向量检索（Dense Retrieval）已经越来越难以满足高精度业务场景的需求。这一点在国家标准类文档知识库（GB/T、GB、行业规范）场景下尤为突出。

标准文档有其鲜明的结构特征：大量标准编号（如 GB/T 17775-2024）、条款编号（如 5.3.2）、专有名词、固定术语、精确数字阈值以及标准间的引用关系。这类内容的核心特点是：语义相似不等于关键信息匹配。纯向量检索擅长捕捉语义关联，却常常丢失编号、数字、精确术语等关键证据，这也是很多 RAG 系统”答非所问”的核心原因。

混合检索（Hybrid Search）正是解决这一问题的主流方案：用密集向量捕捉语义，用 BM25 等稀疏检索算法锁定关键词，两者互补，共同提升召回准确率。

一、为什么标准文档不能只用向量检索

传统 RAG 的检索链路非常直接：用户问题经过 Embedding 模型向量化后，直接在向量数据库中做相似度检索，返回 TopK 结果送入大模型生成答案。

这条链路在通用知识库中尚可运行，但面对标准文档时会出现明显的精度下降。举一个典型例子：

用户提问：GB/T 17775-2024 中 A 级景区厕所数量要求是多少？

纯向量检索返回的结果大概率是：

  • 旅游厕所建设应满足游客需求
  • 景区公共服务设施应完善
  • 卫生设施布局应合理

这些结果在语义上确实相关，但并没有命中真正包含答案的条款：

5.4.3 A级旅游景区厕所数量应满足：高峰期游客需求，每500人至少配置1个厕位

问题出在向量模型的匹配逻辑上。它关注的是”厕所、景区、游客、设施”等高维语义特征，认为”旅游厕所建设要求”与”厕所数量配置要求”语义接近，却无法对 5.4.3、500人、厕位、A级 这些精确字段赋予足够高的权重。

换句话说，向量检索懂”意思”，但不懂”精确匹配”。而标准文档的检索恰恰高度依赖编号、术语和数字的精准命中。

二、混合检索的核心架构：Dense + Sparse

行业内的成熟解法是采用密集向量检索 + 稀疏检索的双路召回架构。一路负责语义理解，一路负责关键词匹配，两路结果融合后再做精排。

密集向量（Dense Vector）负责捕捉语义相关性，保证召回的广度；稀疏检索（Sparse Retrieval，这里以 BM25 为代表）负责字面精确匹配，保证关键信息的召回深度。两者结合，既能够理解用户的自然语言提问，又不会丢失标准文档中的编号、术语、数字等关键证据。

三、密集向量检索（Dense Embedding）

3.1 基本原理

密集向量检索的核心是将文本通过 Embedding 模型映射为固定维度的稠密向量。例如一段”景区厕所建设要求”的文本，会被转换为一个 768 维（或其他维度）的浮点数向量：

在 Milvus 中，向量与原始文本、元数据一起存储在 Collection 中，典型的字段结构包括：

  • 文档 ID
  • 原文内容
  • 向量字段（float vector）
  • 元数据（标准编号、章节、发布年份等）

3.2 相似度计算

向量检索最常用的相似度度量方式是余弦相似度（Cosine Similarity），衡量两个向量在高维空间中的夹角大小。

例如查询向量 Q = [0.1, 0.3, 0.8]，文档向量 D = [0.2, 0.4, 0.7]，计算得到余弦相似度约为 0.95，即认为两者高度相关。

向量检索的优势在于能够跨越字面差异，识别语义上的等价表达，这也是 RAG 能够理解自然语言提问的基础。但它的短板同样明显：对精确关键词、编号、数字的敏感度不足。

四、BM25 稀疏检索的核心思想

BM25 是经典的概率检索模型，也是 Elasticsearch、Lucene 等搜索引擎的核心排序算法。与向量检索不同，它不试图理解语义，只做字面关键词的匹配与打分。

例如查询 “GB/T 17775-2024 5.4.3” 时，BM25 会在所有文档中寻找 GB/T、17775、2024、5.4.3 这些词条的出现情况，并根据词频、稀有度、文档长度综合打分。

BM25 的打分逻辑主要由三部分构成：

4.1 TF（词频，Term Frequency）

词条在文档中出现的次数越多，相关性得分越高。例如一篇反复出现”厕所”的文档，在检索”厕所”相关内容时得分更高。

4.2 IDF（逆文档频率，Inverse Document Frequency）

稀有词条的权重更高。在一个包含一万篇标准文档的知识库中，”景区”可能出现在所有文档里，权重很低；而 “GB/T 17775-2024” 只出现 20 次，它的出现对相关性的贡献就大得多。

4.3 文档长度归一化

长文档天然包含更多词条，容易在词频上占优。BM25 引入了文档长度归一化因子，避免长文档天然获得不合理的高分。

五、Milvus 为什么推荐 Dense + BM25

Milvus 从 2.4 版本开始正式支持稀疏向量（Sparse Vector），这使得在同一个向量数据库内完成双路检索成为可能。

传统的混合检索方案往往需要同时维护两套系统：一套向量数据库做语义检索，一套 Elasticsearch 做关键词检索，架构复杂且维护成本高。而 Milvus 的稀疏向量能力，允许将 BM25 计算得到的稀疏权重直接存储为稀疏向量字段，与密集向量共存于同一个 Collection 中。

两者的存储形式对比如下：

  • 密集向量：稠密的浮点数数组

  • 稀疏向量：键值对形式，只记录非零维度的词条与权重

七、三种结果融合策略

两路召回的结果如何合并，直接影响最终检索效果。行业内主流的融合方式有三种。

7.1 加权打分融合（Weighted Score）

最直观的融合方式，对密集向量得分和 BM25 得分分别赋予权重后相加。

例如密集得分为 0.85，BM25 得分为 0.60，取 α = 0.7，则最终得分为：

α 取值可以根据业务场景调整：语义为主的场景调高 α，关键词为主的场景调低 α。这种方式实现简单，适合大多数通用知识库。

7.2 倒数排名融合（RRF, Reciprocal Rank Fusion）

RRF 是企业级搜索更推荐的方案。它不直接比较两路的分数，而是基于排名进行融合。

其中 r 是文档在对应召回列表中的排名，k 是常数（通常取 60）。

举一个简化的例子：

  • 向量检索排名：文档A 第1，文档B 第3
  • BM25 检索排名：文档C 第1，文档A 第2

文档 A 的 RRF 得分为 1/(60+1) + 1/(60+2)，在三份文档中最高。

RRF 的最大优势是不需要对不同模型的分数做归一化。向量分和 BM25 分的量纲、分布完全不同，直接加权很容易被某一路主导；而 RRF 只关心排名，稳定性更强，是工业界的首选方案。

7.3 学习排序（Learning To Rank）

通过训练排序模型，基于查询、文档特征和用户点击/标注数据学习最优排序方式。Bing、Google、电商搜索等大规模搜索系统普遍采用。

但对于大多数 RAG 场景而言，标注数据量有限，训练和维护成本高，投入产出比不高，一般不做首选。

八、国家标准知识库的落地实践

回到国家标准文档这个具体场景，混合检索的落地需要在字段设计、切片策略、查询处理三个环节做针对性优化。

8.1 文档字段设计

8.2 Chunk 切片策略

标准文档不适合使用普通的固定长度切片（如每 500 字一刀切）。标准有天然的章节层级结构，应当按条款/章节切分，保证每个切片对应一条完整的条款内容。

例如：

每个切片对应一个最小条款单元，既保证了语义完整性，也让条款编号与正文在同一个向量空间内，提升检索的精准度。

8.3 查询改写（Query Rewrite）

用户的自然语言提问往往缺少标准编号、术语等关键信息，直接检索效果有限。可以在检索前增加一步查询改写，由大模型对原始问题进行扩展，补充标准编号、相关术语、可能的章节等信息。

例如原始问题”A级景区厕所要求？”，改写后可能扩展出：

  • 相关标准：GB/T 17775
  • 关键词：A级旅游景区、厕所、公共卫生设施、数量、配置要求
  • 可能章节：5.4.3

用改写后的查询再做混合检索，召回质量会有明显提升。

总结

密集向量检索负责”理解意思”，BM25 负责”找准关键词”，Milvus 的混合检索能力将两者在同一系统内结合，让 RAG 既能理解用户的自然语言提问，又不会丢失国家标准里的编号、术语、数字等关键证据。

对于国家标准知识库这类对精确性要求高、结构化特征明显的场景，Dense Embedding + BM25 + RRF 融合 + BGE-Reranker 重排是目前最接近工业级搜索引擎水准的落地方案，在保证工程复杂度可控的前提下，能够显著提升 RAG 的回答准确率。
