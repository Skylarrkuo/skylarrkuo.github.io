写了几年业务代码，发现数据结构早就藏在每一行里

说实话，刚工作那会儿，我一直觉得”数据结构与算法”是面试八股文，跟写CRUD没什么关系。

直到某天我看到同事把一个嵌套 for 循环（O(n²)）改成了一行 stream().collect(Collectors.toMap(...))，数据量一大，接口响应直接从 3 秒降到了 50ms。

从那以后我就开始留意——平时写的业务代码里，到底哪些地方其实在用数据结构？

这篇文章不讲八股，不刷 LeetCode，就聊聊那些你天天写、可能没意识到自己在用的玩意儿。

一、HashMap — 真·占了 80% 场景

我统计了一下自己近半年的提交记录，HashMap 出现的频率遥遥领先。不夸张，真的是日常开发的第一生产力。

1. 别再嵌套循环查数据了

这个场景太常见了：你从 A 表查出一批数据，要跟 B 表的数据关联。很多人的第一反应是两个 for 循环套一起，但如果数据量稍微大一点，性能就扛不住。

这玩意儿我几乎每个接口都在用。尤其当你从 ES、RPC 或者数据库里拿到一批数据，要做关联、要做补全信息的时候，先转 Map 再操作，基本是标准操作。

2. 分组统计，一行搞定

groupingBy 这个东西，写过报表类需求的人肯定不陌生。什么”按渠道统计”、”按地区汇总”，本质都是在分组。

3. 计数器模式

2. 多字段排序

六、双指针 / 滑动窗口 — 数据处理利器

1. 合并两个有序列表

归并排序的 merge 步骤，直接就能用来合并两个有序数据源。日志合并、时间线合并、多数据源同步……都能用。

2. 滑动窗口：最近 N 天统计

做数据大屏或者监控看板的时候，”最近 N 天的平均值”这种需求很常见。滑动窗口的好处是 O(n) 搞定，不用每次重新算。

七、LRU 缓存 — 连面试官都爱考，业务里也真用

用起来非常简单：

Java 的 LinkedHashMap 天然支持 LRU，重写一个方法就完事了。面试的时候手写 LRU 需要自己维护双向链表 + HashMap，但业务里真不用那么折腾。

什么时候会用到？调用频率高、数据量不大的热点数据缓存，比如用户信息、配置项、字典值。比直接打 DB 快一个量级。

八、图 — 关系网络、社交推荐

这个在社交场景里是刚需——“你可能认识的人”、”二度好友”。本质就是图的 BFS 遍历，限制一下层数就行。

非社交场景也有：依赖关系分析（A 依赖 B、B 依赖 C）、流程编排（任务之间的先后关系），都是图的问题。

九、位运算 — 权限系统最爱

一个 int 就能存几十个权限标记，判断、组合、移除全是位运算，又快又省空间。Linux 的文件权限（rwx）就是这个思路。

最后

写这篇文章的目的不是让大家回去背算法——是帮你在遇到业务场景的时候，能快速反应过来”这个东西用什么数据结构最合适”。

一张速查表放在这，收藏了不亏：

业务场景 该想到的数据结构/算法 时间复杂度
快速查找、ID 映射 HashMap O(1)
分组统计、词频计数 HashMap + groupingBy/merge O(n)
撤销/重做、路径回退 栈 (Deque) O(1)
异步消费、任务排队 队列 (Queue) O(1)
优先级任务调度 优先队列 (PriorityQueue) O(log n)
菜单/组织/地区树 树 + 递归 + HashMap O(n)
最近 N 条、滑动均值 滑动窗口 / 双指针 O(n)
热点数据缓存 LRU (LinkedHashMap) O(1)
社交关系、依赖分析 图 + BFS/DFS O(V+E)
权限标记 位运算 O(1)
排行榜、TopK 堆 / 排序 O(n log k)

总结：HashMap 能解决大部分问题，树和排序解决剩下的大部分，剩下的一小撮才需要图、位运算这些。

先把这些用熟，比刷 200 道 LeetCode 管用。
